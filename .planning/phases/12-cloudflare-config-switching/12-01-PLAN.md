## Artifacts this plan produces

- Types/functions: `TargetPublicRuntimeConfig`, `AuditedPublicRuntimeOverlay`, `AuditedPublicRuntimeInput`, `TargetDeployProjection`, `TargetWorkflowProjection`, `TargetPagesBuildEnv`, `TargetPagesSurface`, `TargetMutationPreparation`, `TargetRemoteEntry`, `buildTargetProjections`, `parseAuditedPublicRuntimeInput`, `assertPublicRuntimeConfig`, `prepareTargetMutation`, `runPreparedTargetMutation`, `materializeTargetDeployConfig`, `parsePagesBuildEnv`, and explicit allowlists for Vite/Nuxt public keys, app base paths, direct remote entries, and child environment keys. `TargetPagesSurface` is imported from the profile schema's closed `targetPagesSurfaceValues` vocabulary (`dashboard`, `auth`, `blog`, `movie`, `comic`, `tavern`), never inferred from a Pages project string.
- CLI contract: `pnpm target-profile` has a mandatory-`--target` projection command and three disjoint mutation entry paths. `target-deploy` is local-only (`scope=local`): it performs local projection/Wrangler-profile preflight then materializes once for an interactive deploy. `prepare-mutation --scope ci` is CI-only: it resolves, preflights, live-checks, and materializes exactly once, then appends only allowlisted scalar outputs to `GITHUB_OUTPUT`; it never invokes local projection checks, reads `.dev.vars`/`.env`, accepts a Wrangler profile, or dumps `process.env`. `run-prepared-entry --entry <closed-entry> --prepared-context <exact-path>` is CI-only and re-parses the exact preparation output before running only its registered fixed D1/crawler operation through a controlled executor; it rejects caller target/resource/API/project/build values, a context outside its validated run boundary, and arbitrary child argv. `target-remote-entry --target <id> --entry <closed-entry>` is the operator/package-only remote path: it calls the same exported `prepareTargetMutation` contract with `scope=remote`, then reaches that same controlled executor. It rejects CI scope, `--skip-preflight`, `--no-preflight`, arbitrary child argv, caller resource/API/project/build values, and ambient target identity before preparation. A Pages `prepare-mutation` is discriminated as `--command pages-deploy` or `--command pages-rollback` plus required `--surface <TargetPagesSurface>`; generic `--command deploy` is invalid for Pages. The supplied surface is validated against the selected profile before materialization and is the sole selector for `pages_project` and `pages_build_env_path`; callers can never pass a project name, a build-env map, or an output path.
- Deploy templates: target-neutral committed API/gateway/dashboard Wrangler templates. A generated API/gateway Worker config is a unique temporary file directly in that app's existing config directory (for example `apps/api/.target-wrangler.<run-id>.toml`), so `main = "src/index.ts"` and API `migrations_dir = "../../packages/db/drizzle"` retain their established relative resolution. Callers receive the absolute generated path and must pass it as separate `--config <path>` argv tokens to every config-capable Wrangler deploy/D1/R2/Worker-rollback command; Pages deploy is the explicit Wrangler CLI exception because `wrangler pages deploy` has no `--config` option and instead receives validated projected `--project-name` plus generated public build env.
- Direct local deploy entry points: root `target-deploy` and API/gateway/dashboard/blog `deploy` scripts are `scope=local` only, require explicit `--target`, and run local preflight/materialization exactly once; every local Pages invocation also requires a closed `--surface`, which is selected by the package wrapper rather than a project literal. CI never calls this wrapper. The existing API/gateway `dev` scripts remain byte-for-byte unchanged local-development commands.
- Direct remote package entry foundation: `TargetMutationPreparation` is the only runtime handoff from `TargetResolution` to a D1 migration/cleanup or crawler child. It contains selected-profile non-secret identity, run-scoped materialized config/context paths, and a closed operation identity, never caller-provided API/D1/R2/account/project values. CI exposes only the validated run-scoped `prepared_context_path` alongside its existing fixed output keys; it is re-parsed/owned by the selected run before a fixed prepared-entry operation may consume it. The helper constructs a fresh child environment from only command-runtime keys, selected profile API/R2/D1 identity, and fixed standard secret names whose presence/account match has already passed preflight. `DATABASE_URL`, `API_URL`, `R2_BUCKET_NAME`, `R2_PUBLIC_URL`, `R2_ACCOUNT_ID`, `STARYE_TARGET_ID`, and other target-identifying ambient values are never forwarded or used to choose a target; ambient identity causes blocking failure (standard `CLOUDFLARE_ACCOUNT_ID` is compare-only during preflight and the selected profile value is what reaches the child). The implementation remains import-safe and injects a fakeable executor for tests.
- Gates: CI/remote preflight skips operator-owned local env projection checks, retains exact Environment/account/credential checks, and adds redacted read-only selected Worker/Pages checks alongside D1/R2/KV checks. Pages checks compare the profile-selected project name to the actual target-account project and block on absence or mismatch; no human checkpoint or project-creation fallback is permitted.
