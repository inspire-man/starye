## Workflow contract used by every task

1. Determine an explicit tracked target: required `workflow_dispatch` input, or a checked-in literal target for push/schedule/pull-request triggers.
2. Run the non-secret resolver job using `pnpm target-profile` output; it resolves only target ID and mapped `githubEnvironment`, receives no deployment secrets, and does not materialize configuration.
3. Bind the mutation job to the resolver-derived `githubEnvironment`, then inject fixed standard secret names only. A dispatch input cannot select/override the Environment and YAML never uses computed secret names.
4. Exactly once in each mutation job, invoke `pnpm target-profile prepare-mutation --scope ci --target <resolved> --ci-environment <mapped> --command <worker-deploy|migrate|remote-crawl|cleanup|worker-rollback> --github-output "$GITHUB_OUTPUT"` after secrets are injected for a non-Pages operation. A Pages deployment instead invokes `pnpm target-profile prepare-mutation --scope ci --target <resolved> --ci-environment <mapped> --command pages-deploy --surface <dashboard|auth|blog|movie|comic> --github-output "$GITHUB_OUTPUT"`; a Pages rollback invokes the same command with `--command pages-rollback --surface "${{ inputs.pages_surface }}"`. `--surface` is parsed from the profile schema's `TargetPagesSurface` vocabulary, not from a project literal, and is mandatory for every Pages path. This one CI-only entry resolves, validates the selected surface/project, runs CI preflight and read-only selected Worker/Pages/D1/R2/KV checks, and materializes. It must not run `target-deploy`, local projection checks, read local `.dev.vars`/`.env`, pass `--env-root`/`--wrangler-profile`, accept a project name/build path, or materialize a second time.
4a. For a Pages build, the mutation job supplies the exact fixed non-secret `STARYE_PUBLIC_*` variables from its resolver-mapped GitHub Environment to the Plan 12-01 overlay loader before `id: prepare`; the loader accepts only `STARYE_PUBLIC_SENTRY_DSN`, `STARYE_PUBLIC_SENTRY_RELEASE`, `STARYE_PUBLIC_BUILD_MODE`, `STARYE_PUBLIC_MONITORING_ENABLED`, `STARYE_PUBLIC_FEATURE_ARIA2`, `STARYE_PUBLIC_FEATURE_ARIA2_WS`, `STARYE_PUBLIC_FEATURE_RATING`, `STARYE_PUBLIC_FEATURE_AUTO_SCORE`, and `STARYE_PUBLIC_FEATURE_PERF_MONITOR`. It parses them field-by-field into the run-scoped `AuditedPublicRuntimeInput` and never forwards the `STARYE_PUBLIC_*` names, an arbitrary environment map, a `secrets.*` value, a target/resource identifier, or a raw VITE/Nuxt value to the build. Local `target-deploy` uses the same named overlay loader from its controlled local input. Missing optional Sentry values are allowed; all other missing/invalid/unknown overlay values block before a Pages build.
5. The prepare step is named `prepare` and its fixed output keys are consumed verbatim: API Worker/D1/R2 migration commands use `${{ steps.prepare.outputs.api_config_path }}`, gateway Worker commands use `${{ steps.prepare.outputs.gateway_config_path }}`, and Worker rollback chooses only the matching one. Each occurs as exact separate argv tokens `--config`, that output expression; YAML may not recompute, copy, or rewrite a path. `wrangler pages deploy` is the documented exception because its current CLI has no `--config`. Before it, every Pages workflow invokes exactly `pnpm target-profile run-pages-build --surface <its-fixed-surface> --pages-build-env-path "${{ steps.prepare.outputs.pages_build_env_path }}"`; that command re-parses the generated dotenv with the surface/framework allowlist, adds only the validated values to a `shell: false` child process, and runs the exact fixed app build (`pnpm --filter dashboard build`, `pnpm --filter starye-auth build`, `pnpm --filter blog build`, `pnpm --filter @starye/movie-app build`, or `pnpm --filter @starye/comic-app build`). The deploy then uses only `--project-name "${{ steps.prepare.outputs.pages_project }}"`. Blog therefore deploys the resolved `blog-pages` value without a YAML literal, and a missing/unknown surface or live-project mismatch blocks before build, deploy, or manual guidance.
6. Remove only the prepare-generated files in an `always()` cleanup step, including the selected `pages_build_env_path`, and report target/resource/run identity only. Neither prepare nor the build step may use `set -x`, `cat`, `printenv`, `env`, or an echo/log of parsed dotenv values. Pages rollback remains `exit 1`, but only after its surface-specific prepare succeeds; its manual console instruction names `${{ steps.prepare.outputs.pages_project }}`. Cleanup still runs after that failure. Do not create Pages projects, emit token values, add a target selector UI, or edit RUNBOOK in this phase.
7. CI executes D1/crawler work only through the Plan 12-01 prepared-entry registry. After CI `prepare`, migration, crawler, and monthly-cleanup jobs call the fixed `pnpm target-profile run-prepared-entry --entry <closed-entry> --prepared-context <exact-output>` command; no inline `wrangler d1 --remote`, direct `tsx` mutation script, `DATABASE_URL`, `API_URL`, R2/D1 name, account, bucket, or target identity may replace it. The registry validates a run-scoped context path and starts a fresh allowlisted child environment. Direct package wrappers and the independent crawler-source audit are deliberately deferred to Plan 12-04, where the same registry must close those non-CI paths.
