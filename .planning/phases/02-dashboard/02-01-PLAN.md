<objective>
在 Better Auth session 响应中注入 `githubId` 字段，为 Gateway 白名单判断和 API requireAuth 短路提供数据基础。

Purpose: D-01/D-04 要求 Gateway 和 API 通过 `ADMIN_GITHUB_ID` env var 判断白名单，但当前 `SessionUser` 类型没有 `githubId` 字段。GitHub ID 存储在 `account` 表的 `accountId` 字段（`providerId='github'`）。本 plan 通过 Better Auth `additionalFields` + `callbacks.session` 机制从 `account` 表查询并注入 `githubId`，使下游 Plan 02（API 白名单短路）和 Plan 03（Gateway 拦截）能够读取此字段。

Output: `SessionUser` 类型包含 `githubId?: string`，`/api/auth/get-session` 响应 JSON 包含 `user.githubId`，单元测试验证字段存在且值正确。
</objective>

callbacks: {
  session: async ({ session, user }: { session: schema.Session, user: schema.User }) => {
    // 从 account 表查询 GitHub ID（providerId='github' 的 accountId）
    const db = createDb(env.DB)  // env 已在闭包中可用

        eq(a.userId, user.id),
        eq(a.providerId, 'github')

    })

      session,
      user: {
        ...user,
        isAdult: !!user.isAdult,
        isR18Verified: !!user.isR18Verified,
        githubId: githubAccount?.accountId ?? null,  // 新增
      },
    }
  },
}

**新建 apps/api/src/lib/__tests__/auth.test.ts：**

Better Auth 的 `callbacks.session` 在内部执行，难以直接 mock 整个 Better Auth 实例。最实用的方案是将核心注入逻辑提取为纯函数 `injectGithubIdIntoSession`，单独测试该纯函数，再在 `callbacks.session` 中调用它。

**步骤 1：** 在 `apps/api/src/lib/auth.ts` 中，将 `callbacks.session` 的 githubId 查询逻辑提取为可导出的纯函数（在 `createAuth` 工厂函数外部定义）：

```typescript
// apps/api/src/lib/auth.ts — 在 createAuth 函数外部，文件顶部附近
/**
 * 将 GitHub ID 注入 session user 对象。
 * 提取为纯函数以便单元测试。
 * @param userId - Better Auth user.id
 * @param accounts - account 表查询结果（providerId='github' 的记录）
 */
export function injectGithubIdIntoSession(
  userId: string,
  githubAccount: { accountId: string } | undefined,
): string | null {
  return githubAccount?.accountId ?? null
}
```

然后在 `callbacks.session` 中调用：
```typescript
callbacks: {
  session: async ({ session, user }) => {
    const db = createDb(env.DB)
    const githubAccount = await db.query.account.findFirst({
      where: (a, { and, eq }) => and(
        eq(a.userId, user.id),
        eq(a.providerId, 'github'),
      ),
    })
    return {
      session,
      user: {
        ...user,
        isAdult: !!user.isAdult,
        isR18Verified: !!user.isR18Verified,
        githubId: injectGithubIdIntoSession(user.id, githubAccount ?? undefined),
      },
    }
  },
}
```

import { injectGithubIdIntoSession } from '../auth'

describe('injectGithubIdIntoSession', () => {
  it('有 GitHub account 时返回 accountId', () => {

})

|----------|-------------|

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Elevation of Privilege | Better Auth session callback | mitigate | 查询 account 表时使用 Drizzle ORM 参数化查询（`eq(a.userId, user.id)`），避免 SQL 注入 |
| T-02-08 | Information Disclosure | githubId 字段泄漏 | accept | githubId 是公开信息（GitHub profile 可见），泄漏风险低；仅在已登录 session 中暴露 |
| T-02-09 | Denial of Service | account 表查询性能 | mitigate | `account` 表有 `userId` 索引（schema 定义），查询性能可接受；每次 session 查询只执行一次 |
