## Context

当前爬虫系统存在以下问题：
1. **配置传递 Bug**：`scripts/run-optimized.ts` 使用 `process.env.MAX_MOVIES || '50'` 导致工作流配置（200）被覆盖；`ProgressMonitor` 在初始化时使用硬编码 100 而非配置值。
2. **缺乏增量能力**：每次爬取都全量扫描，已存在内容通过单次查询跳过，浪费 API 调用和时间（如漫画爬取 59/112 处理率）。
3. **测试覆盖不足**：批量状态查询接口 `/api/admin/movies/batch-status` 曾因路由和认证问题返回 401，CI 中无 E2E 测试覆盖，问题需手动发现。

现有架构：
- 批量状态查询 API 已实现并修复（位于 `apps/api/src/routes/admin-movies.ts`）
- 爬虫使用 Puppeteer 和 `got` HTTP 客户端
- GitHub Actions 工作流调度爬虫，通过环境变量传递配置
- 爬虫通过 `ApiClient` 与后端 API 交互，使用服务令牌认证

## Goals / Non-Goals

**Goals:**
- 修复配置传递 Bug，确保环境变量正确传递到爬虫和进度监控
- 实现批量状态查询增量逻辑，爬取前过滤已存在内容
- 添加完整的 E2E 测试覆盖，验证批量查询、增量过滤、配置传递
- 改进进度监控，显示增量统计（已存在/新增比例）

**Non-Goals:**
- 不修改批量状态查询 API（已修复完成）
- 不改变爬虫的并发控制或延迟策略
- 不实现章节级别的批量查询（漫画爬虫章节仍使用单次查询）

## Decisions

### Decision 1: 移除代码内默认值，强制环境变量校验

**选择**：从 `scripts/run-optimized.ts` 移除 `process.env.MAX_MOVIES || '50'` 的默认值逻辑，改为在环境变量未设置时抛出错误并退出。

**理由**：
- 默认值会掩盖配置错误，导致工作流配置（200）被覆盖为代码默认值（50）
- 显式错误更容易发现配置问题，符合 "fail fast" 原则
- 工作流中明确设置环境变量是最佳实践

**替代方案**：
- 保留默认值并添加警告日志 → **拒绝**：警告容易被忽略，问题仍会潜伏
- 使用配置文件而非环境变量 → **拒绝**：增加复杂度，环境变量更适合 CI/CD

### Decision 2: 在爬虫发现阶段后、详情页爬取前执行批量查询

**选择**：在 `OptimizedCrawler.run()` 中，获取电影列表后立即调用批量状态查询，过滤已存在内容，然后进入详情页爬取循环。

**理由**：
- 最小化 API 调用：单次批量查询替代 N 次单次查询
- 尽早过滤减少无效工作：避免详情页爬取、图片下载、数据处理等开销
- 逻辑清晰：增量逻辑集中在一处，易于测试和维护

**替代方案**：
- 在详情页爬取时逐个查询 → **拒绝**：性能差，API 调用量大
- 使用缓存层避免重复查询 → **拒绝**：增加复杂度，批量查询已足够高效

**实现细节**：
```typescript
// 伪代码示例
async run() {
  const discoveredMovies = await this.discoverMovies() // 发现阶段
  const codes = discoveredMovies.map(m => m.code)
  const existingCodes = await this.apiClient.batchQueryMovieStatus(codes) // 批量查询
  const newMovies = discoveredMovies.filter(m => !existingCodes.has(m.code)) // 过滤
  
  console.log(`发现 ${discoveredMovies.length} 部电影，其中 ${existingCodes.size} 部已存在，将爬取 ${newMovies.length} 部新电影`)
  
  for (const movie of newMovies) {
    await this.crawlMovieDetail(movie) // 详情爬取
  }
}
```

### Decision 3: 修复 ProgressMonitor 初始化逻辑

**选择**：修改 `ProgressMonitor` 构造函数调用，传入 `config.limits.maxMovies` 而非硬编码 100。

**理由**：
- 硬编码值与实际配置不一致，导致进度条显示错误（38/100 而非 38/200）
- 进度条应反映实际爬取目标，帮助监控任务完成度

**替代方案**：
- 动态更新进度条总数 → **拒绝**：实现复杂，初始化时传入正确值更简单

### Decision 4: 使用 Vitest + Mock HTTP 实现 E2E 测试

**选择**：使用 Vitest 的 `vi.fn()` mock `fetch` API，模拟批量查询响应，验证爬虫增量逻辑。

**理由**：
- Vitest 已是项目测试框架，无需新增依赖
- Mock HTTP 可控制测试场景（成功、401、超时等），无需真实 API 环境
- 测试可在本地和 CI 中快速运行

**替代方案**：
- 使用真实 API 环境 → **拒绝**：依赖外部环境，测试不稳定且慢
- 使用 Playwright 进行浏览器级别 E2E → **拒绝**：过度复杂，单元/集成测试足够

**测试覆盖范围**：
- `ApiClient.batchQueryMovieStatus` 方法：成功、空列表、401、超时
- `OptimizedCrawler` 增量逻辑：过滤已存在、全部已存在、无已存在
- 配置传递：环境变量正确读取、缺失时错误
- `ProgressMonitor` 初始化：使用配置值而非硬编码

### Decision 5: 漫画爬虫复用批量查询逻辑

**选择**：在 `ComicCrawler` 中添加类似的批量状态查询和过滤逻辑，但章节级别仍使用单次查询。

**理由**：
- 漫画爬虫同样需要增量能力，避免重复爬取已存在漫画
- 章节数量远大于漫画数量，批量查询收益不大，保持现有逻辑

**替代方案**：
- 实现章节级别批量查询 → **延后**：需修改 API，当前不是优先级

## Risks / Trade-offs

### Risk 1: 批量查询 API 失败导致爬虫退出

**风险**：如果批量状态查询 API 返回错误（如 500、超时），爬虫会因异常退出，导致当日爬取失败。

**缓解措施**：
1. 添加重试逻辑（最多 3 次，指数退避）
2. 失败时回退到全量爬取模式（不过滤，依赖单次查询跳过）
3. E2E 测试覆盖 API 失败场景，确保回退逻辑生效

### Risk 2: 环境变量校验过于严格

**风险**：移除默认值后，如果本地开发忘记设置环境变量，脚本会立即失败。

**缓解措施**：
1. 在 README 和错误信息中明确说明必需的环境变量
2. 提供 `.env.example` 文件作为参考
3. 本地测试时可临时降低配置要求（如 `MAX_MOVIES=10`）

**权衡**：牺牲一些开发便利性，换取配置错误的早期发现。

### Risk 3: 测试覆盖率目标过高

**风险**：要求 90%+ 覆盖率可能导致过度测试，编写无意义的测试用例。

**缓解措施**：
1. 聚焦关键路径：批量查询、增量过滤、配置传递
2. 不强求覆盖所有边界情况（如日志格式、进度条样式）
3. 覆盖率报告仅作参考，核心是测试关键场景

**权衡**：接受部分辅助代码（如日志、格式化）未完全覆盖，确保核心逻辑测试充分。

## Migration Plan

### Phase 1: 修复配置传递（低风险）

1. 提交代码变更：移除默认值，添加环境变量校验
2. 在工作流中验证 `MAX_MOVIES` 和 `CRAWLER_MAX` 已正确设置
3. 运行一次测试爬取，确认配置生效

**回滚**：如失败，回退到使用默认值的版本

### Phase 2: 实现增量逻辑（中等风险）

1. 实现批量查询和过滤逻辑
2. 本地测试，验证增量过滤正确（使用小数据集）
3. 部署到 CI，运行一次爬取，观察日志输出
4. 对比前后爬取时间和内容数量，验证效率提升

**回滚**：如批量查询失败，临时禁用增量逻辑，回退到全量爬取

### Phase 3: 添加 E2E 测试（低风险）

1. 编写测试套件，覆盖关键场景
2. 在 CI 中运行测试，确保通过
3. 添加测试覆盖率检查，要求核心逻辑达到目标

**回滚**：无需回滚，测试失败会阻止部署

### Phase 4: 改进进度监控（低风险）

1. 更新进度输出，显示增量统计
2. 验证输出格式清晰易读
3. 观察几次爬取，确认统计准确

**回滚**：如输出格式有问题，调整格式字符串

## Open Questions

1. **批量查询性能**：当前单次查询最多支持多少代码？是否需要分批查询？
   - **答案**：暂无明确限制，先实现单次查询所有代码，如遇性能问题再优化（如分批 100 个）

2. **章节级别批量查询**：是否需要在未来支持漫画章节的批量查询？
   - **答案**：延后，当前优先解决漫画和电影的爬取效率问题

3. **增量统计持久化**：是否需要保存每次爬取的增量命中率，用于长期分析？
   - **答案**：延后，当前仅输出到日志，未来可考虑存储到数据库
